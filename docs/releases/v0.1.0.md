# kotidy v0.1.0

## Initial release

A real Gradle plugin for RSpec-style nested test output over Kotest's
`DescribeSpec` -- hooks Gradle's own `TestListener` API directly and walks
the real `TestDescriptor.parent` chain, rather than parsing raw text output
the way [`gorderly`](https://github.com/woodie/gorderly) (Go) and
[`xctidy`](https://github.com/woodie/xctidy) (Swift) have to. Extracted from
the byte-for-byte-copied `TestListener` block `next-caltrain-kotlin`/
`humane-kotlin`/`huck` each carried in their own `build.gradle.kts`.

## Usage

```kotlin
plugins {
    id("com.netpress.kotidy")
}

kotidy {
    style = "fd" // classic (default), fd, fs, or fv
}
```

Or override per-invocation: `./gradlew test -Dkotidy.style=fd`.

## Styles

Four named styles, matching the shared surface `gorderly`/`xctidy` already
document: `classic` (glyph + per-test elapsed time), `fd` (RSpec's
documentation format), `fs` (Mocha's spec format), and `fv` (Vitest's own
tree, with its `Test Files`/`Tests`/`Duration` footer). See README's style
table.

## Installation

No Gradle Plugin Portal publish yet -- consumed as a Gradle composite build
(`pluginManagement { includeBuild("../kotidy") }`), the same pattern `huck`
uses for `humane-kotlin`. Publishing to the Portal is set up
(`com.gradle.plugin-publish` applied, metadata configured) but the actual
`publishPlugins` run and portal approval are still pending -- see
`docs/COWORK.md`.

## Testing

`Styles.kt` (the pure formatting logic, no Gradle API) is unit-tested with
Kotest's `DescribeSpec`. `KotidyPlugin.kt`'s `TestListener` wiring itself
isn't -- verifying it means applying it in a real consumer and reading
actual console output, which `humane-kotlin`'s own `make test` now
confirms end to end.

## Scope

Consumed by `next-caltrain-kotlin`, `humane-kotlin`, and `huck` via the
composite-build pattern. `humane-kotlin`'s CI workflow is the only one of
the three wired up to build against `kotidy` so far.
